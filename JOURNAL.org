** Decisions
*** [2026-04-02] – Chose SQLite + Tkinter (Approach 1) over PySide6 or local web app
**** Reason
Fastest path to a working prototype; all dependencies are stdlib or
near-stdlib.  Explicit migration path to PySide6 planned: keep all
non-UI code in separate modules so =ui/= can be replaced independently.

**** Alternatives considered
- PySide6: better UI, but heavier dependency and steeper Qt learning curve
- Local web app (FastAPI + browser): rich UI, but not a true desktop app and
  harder to package

*** [2026-04-02] – Dedup via composite unique constraint, not hashing
**** Reason
=UNIQUE(date, amount, description)= + =ON CONFLICT DO NOTHING= is DB-enforced,
transparent to query, and easier to debug than a hash column.

**** Alternatives considered
Hash of canonical fields: same vulnerability to description variance, but
opaque column values make debugging harder.

*** [2026-04-02] – Description canonicalization required before insert
**** Reason
The composite unique constraint only works if the same transaction always
produces the same description string.  Raw CSV strings are unreliable
(trailing spaces, encoding differences, bank formatting changes).

*** [2026-04-02] – Use =uv= instead of =pip= for dependency management
**** Reason
=uv= provides a proper lock file (=uv.lock= ≈ =package-lock.json=), manages the
venv automatically, and replaces =pip=, =venv=, and =pip-tools= in one tool.
=requirements.txt= has no lock file concept; reproducible installs require pinning
by hand or a separate tool.

**** Alternatives considered
Plain =pip= + =requirements.txt=: simpler but no lock file, non-reproducible builds.

*** [2026-04-02] – Data location via =BANK_ANALYZER_DATA_DIR= env var
**** Reason
Allows dev (=./data=) and prod (OS user-data dir) to differ by one env var,
not by code changes.  =python-dotenv= loads =.env= automatically at startup.

** Decisions
*** [2026-04-07] – Encoding specified per bank in =BANKS= dict, not auto-detected
**** Reason
The bank is known in advance (Polish bank, CP-1250), so runtime detection adds
complexity with no benefit.  Each =BANKS= entry includes an =encoding= key passed
to =open()= / =csv.DictReader=.

**** Alternatives considered
=charset-normalizer= for runtime detection: accurate but an extra dependency,
unnecessary when the encoding is known.

*** [2026-04-09] – Header autodetection via anchor column set; synthetic names for unnamed columns
**** Reason
Banks may prepend unstructured preamble rows (variable count) before the CSV table,
and may rename or reorder named columns across export versions.  Scanning for a row
that contains all anchor column names is resilient to both.  Unnamed columns (empty
header cells) get synthetic names based on the last preceding named column
(e.g. =Opis=, =Opis+1=, =Opis+2=) so =col_map= can reference them stably.

**** Alternatives considered
- Hardcoded =skip_rows= count: breaks silently if preamble length changes.
- Config-as-module (functions per bank): maximum flexibility but end users cannot
  define new banks without writing Python — overkill for this use case.

**** Anchor columns
Derived at runtime from =date_col=, =amount_col=, and the base (non-synthetic) names
in =description_cols=.  Extra named columns not in the anchor set are silently ignored.

*** [2026-04-09] – Decimal and thousand separators explicit in bank config
**** Reason
Banks are inconsistent: some use a period as decimal separator, some a comma, and
thousand-grouping characters vary too.  Tolerating both separators without config
is ambiguous (=1.234= could be 1234 or 1.234).  Explicit per-bank =decimal_sep= and
=thousand_sep= keys remove the ambiguity and keep parsing deterministic.

**** Alternatives considered
Auto-detect based on position of separators: complex, error-prone for edge cases
like round numbers (=1.000=).

*** [2026-04-09] – =amount_regex= in bank config for banks that mix amount and currency in one column
**** Reason
Some banks (e.g. mBank) put amount and currency in the same column (e.g. =-43,52 PLN=).
An optional =amount_regex= key in the bank config holds a pattern with a named capture
group =amount= (e.g. =r'(?P<amount>-?[\d\s,.]+)\s+[A-Z]{3}'=) to extract just the
numeric part.  When absent (e.g. PKO BP), the raw column value is used directly.

**** Alternatives considered
A separate =amount_col= and =currency_col= split: requires the bank to always have
separate columns, which is not universally true.

*** [2026-04-09] – Multiple CSV columns joined into one =description= field
**** Reason
Some banks split transaction details across several columns (e.g. =Opis transakcji=
holds the title/reference while the next unnamed column holds the merchant location).
A =description_cols= list in the bank config names the columns to join; they are
concatenated with a space separator and then passed through =canonicalize_description()=,
so whitespace collapsing handles any padding between joined fragments automatically.

** Decisions
*** [2026-04-13] – Substring match (not regex) for rules
**** Reason
Simpler to enter in a UI; sufficient for typical merchant name matching.  Regex
adds power the user would rarely need and makes rule authoring harder.

*** [2026-04-13] – Rules applied both at import time and on-demand
**** Reason
Automatic application at import is the main convenience; on-demand re-run lets
the user apply newly added rules to already-imported transactions without
re-importing.

*** [2026-04-13] – Categorizer skips already-categorized transactions (MVP)
**** Reason
Preserves manual corrections — if the user overrides a rule by hand, a
subsequent auto-run must not undo that.  Future: add a "recategorize since
date" mode (see deferred task at end of file).

*** [2026-04-13] – Categorization rules are global (not bank-scoped)
**** Reason
Most merchants appear across all banks with similar descriptions.  Bank-scoped
rules would require duplicating every rule per bank — a real usability cost for
a problem most users will never hit.  No =bank= column on
=categorization_rules=, no escape hatch, to keep the MVP simple.

**** Alternatives considered
Optional nullable =bank= FK on =categorization_rules= (global when =NULL=):
ruled out to avoid schema and UI complexity in the MVP.
